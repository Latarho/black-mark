# Black Mark Web

`Black Mark Web` — клиентское демо-приложение для просмотра организационной структуры, оценки сотрудников, работы с матрицами риска/эффективности и управления опросами подразделений.

Проект построен как Next.js приложение с App Router. Основные данные сейчас представлены моками в `lib/`, поэтому приложение можно запускать локально без внешнего API, базы данных и переменных окружения.

## Стек

- `Next.js 16` с App Router и Turbopack в dev-режиме.
- `React 19`.
- `TypeScript 5` со строгой типизацией.
- `Tailwind CSS v4`.
- `shadcn/ui` стиль `radix-mira`, Radix primitives, lucide icons.
- `d3` и `d3-org-chart` для интерактивного графа оргструктуры.
- `Vitest` для unit-тестов чистой логики.
- `ESLint`, `eslint-config-next`, `Prettier` и `prettier-plugin-tailwindcss`.

## Быстрый Старт

Установите зависимости из директории `web`:

```bash
npm ci
```

Запустите dev-сервер:

```bash
npm run dev
```

По умолчанию Next.js поднимет приложение на `http://localhost:3000`.

## Маршруты

| Путь | Назначение |
| --- | --- |
| `/` | Основной экран оценки сотрудников, фильтры, таблица, матрицы, карточка сотрудника, внешние оценки и опросы. |

## Архитектура UI

`app/layout.tsx` подключает шрифты, глобальные стили и `ThemeProvider`.

`app/(app)/layout.tsx` оборачивает все рабочие страницы в `AppShell`.

`components/app-shell.tsx` собирает общий интерфейс:

- `TooltipProvider` для тултипов;
- `SidebarProvider` и `AppSidebar`;
- верхний header с кнопкой sidebar, breadcrumbs и переключателем темы;
- scrollable content area для страниц.

`components/ui/` содержит локальные shadcn/ui компоненты. Это не внешняя библиотека в runtime, а исходный код компонентов внутри проекта. Их можно адаптировать, но изменения стоит делать осознанно, потому что они влияют на все места использования.

`components/detail-card-section.tsx` содержит общий презентационный примитив для секций и полей карточек. Его используют экран оценки и персональный отчёт, чтобы не дублировать одинаковую разметку.

## Доменные Области

### Оценка Сотрудников

Главный экран находится в `app/(app)/page.tsx`.

Он отвечает за:

- фильтрацию сотрудников по подразделениям, ФИО, должностям, критичности, ФКР, компенсации, переработкам и оценкам;
- переключение полной и краткой таблицы;
- локальные override-значения для категории сотрудника и вероятности увольнения;
- отображение матриц `9-box` и `12-box`;
- карточку сотрудника с вкладками профиля, нагрузки, опросов, внешней оценки, РИТМ и компенсации;
- локальный блок заметок руководителя;
- вкладки внешних оценок и опросов.

Чистая логика оценки вынесена в `lib/assessment-model.ts`. Там находятся:

- типы уровней и фильтров;
- подписи и классы тегов;
- расчёт A-E критичности;
- расчёт строк/ячеек матриц;
- бакетизация сотрудников для `9-box` и `12-box`;
- форматтеры времени и даты;
- справочники опций для select/filter UI.

Если меняется бизнес-правило оценки, сначала ищите его в `lib/assessment-model.ts`, а не в JSX страницы.

### Организационная Структура

Демо-дерево подразделений и сотрудники находятся в `lib/bank-org-mock.ts`.

Ключевые сущности:

- `OrgUnit` — узел оргструктуры;
- `StaffMember` — сотрудник и его демо-профиль;
- `ORG_ROOT` — корневой узел дерева;
- `STAFF` — список сотрудников;
- `findUnit`, `collectUnitIds`, `getBreadcrumb`, `getStaffLineManagers` — утилиты обхода дерева.

`components/staff-org-explorer.tsx` показывает оргструктуру как список/детальный просмотр.

`components/d3-org-chart-view.tsx` отвечает за D3-граф. Подготовка плоских строк для графа находится в `lib/org-chart-d3-data.ts`, а HTML-шаблоны узлов и экранирование динамических значений вынесены в `lib/d3-org-chart-node-html.ts`.

Важно: все пользовательские или моковые строки, попадающие в D3 HTML-шаблоны, должны проходить через `escapeOrgChartHtml`. Это сохраняет текущий контракт безопасности для строкового HTML.

### Опросы Подразделений

`components/unit-surveys-panel.tsx` реализует демо-панель опросов:

- создание черновика опроса;
- выбор шаблона;
- выбор подразделения и участников;
- фильтрация по статусам;
- табличный и карточный режимы;
- редактирование деталей.

Шаблоны и статусы опросов находятся в `lib/surveys-manager-mock.ts`.

Общие опции подразделений находятся в `lib/unit-options.ts`. Используйте `collectUnitOptions` вместо локального повторения обхода дерева.

### Внешняя Психодиагностика

Типы JSON-отчёта описаны в `lib/psych-report-types.ts`.

Демо-данные лежат в `lib/psych-report-demo.json`.

Импорт демо-данных проходит через `lib/psych-report-demo-data.ts`, где проверяется базовая top-level структура. Компонент `components/external-assessment-personal-report.tsx` получает уже типизированный `PsychPersonalReport`.

### Презентация Сотрудников

`lib/staff-presentation.ts` содержит форматтеры и визуальные правила:

- ФИО;
- инициалы;
- подписи уровней оргструктуры;
- классы тегов;
- классы fallback-аватаров;
- размеры страниц таблиц.

`components/staff-member-avatar.tsx` использует эти правила для отображения фото или fallback-инициалов.

## Данные И Моки

Сейчас приложение не ходит во внешний backend. Основные источники данных:

- `lib/bank-org-mock.ts` — оргструктура и сотрудники;
- `lib/surveys-manager-mock.ts` — шаблоны и статусы опросов;
- `lib/psych-report-demo.json` — пример персонального отчёта;
- `lib/assessment/nine-box-profiles.ts` — описания профилей для 9-box.

При подключении реального API лучше сохранить текущие типы как контракт UI-слоя и добавить адаптеры данных в `lib/` или отдельный слой `services/`.

## Тесты

Тесты запускаются командой:


Конфигурация Vitest находится в `vitest.config.ts`.

Сейчас покрыты:

## shadcn/ui

Проект настроен через `components.json`:

- style: `radix-mira`;
- RSC: включён;
- Tailwind CSS: `app/globals.css`;
- icon library: `lucide`;
- aliases: `@/components`, `@/components/ui`, `@/lib`, `@/hooks`.

Добавление компонента:

```bash
npx shadcn@latest add button
```

Импорт UI-компонентов:

```tsx
import { Button } from "@/components/ui/button"
```

Правила для новых UI-блоков:

- сначала проверяйте, есть ли подходящий компонент в `components/ui/`;
- используйте семантические токены (`bg-background`, `text-muted-foreground`, `border-border`) вместо произвольных цветов, если нет явной причины;
- для spacing предпочитайте `gap-*`, а не `space-*`;
- для одинаковых ширины/высоты используйте `size-*`;
- для условных классов используйте `cn()` из `lib/utils.ts`;
- для иконок в `Button` используйте `data-icon`, если компонент это поддерживает;
- для overlay-компонентов обязательно сохраняйте доступные title/description.

## Темизация

Тема подключена через `components/theme-provider.tsx` и `next-themes`.

Переключатель темы находится в `components/theme-toggle.tsx`.

Глобальные CSS-переменные и Tailwind v4 theme tokens находятся в `app/globals.css`. Если нужно менять палитру или базовые tokens, делайте это там, а не в отдельных CSS-файлах.

## TypeScript И Импорты

Проект использует alias `@/*`, настроенный в `tsconfig.json`.

Рекомендации:

- импортируйте внутренние модули через `@/`;
- типы, используемые несколькими компонентами, держите в `lib/`;
- UI-компоненты с состоянием, эффектами или browser API должны начинаться с `"use client"`;
- чистую бизнес-логику держите отдельно от JSX, чтобы её можно было тестировать без React.

## Добавление Нового Раздела

1. Создайте страницу в `app/(app)/.../page.tsx`.
2. Добавьте пункт в `lib/nav-config.ts`.
3. Если нужен общий layout, используйте существующий `AppShell`.
4. Для UI сначала проверьте `components/ui/`.
5. Чистую доменную логику вынесите в `lib/` и добавьте тест.
6. Запустите `npm run lint`, `npm run typecheck`, `npm test`.

## Добавление Нового Демо-Поля Сотрудника

1. Добавьте поле в `StaffMember` в `lib/bank-org-mock.ts`.
2. Заполните значение в генераторе демо-профиля или конкретном demo staff override.
3. Если поле влияет на расчёты, добавьте функцию/константу в `lib/assessment-model.ts`.
4. Если поле только отображается, добавьте форматтер в `lib/staff-presentation.ts` при необходимости.
5. Обновите UI в `app/(app)/page.tsx` или связанных компонентах.
6. Добавьте/обновите тесты для расчётов.

## Добавление Нового Правила Оценки

1. Описывайте rule как чистую функцию в `lib/assessment-model.ts`.
2. Не зашивайте бизнес-условия прямо в JSX таблицы или модалки.
3. Добавьте тесты для обычного случая, пограничного случая и отсутствующих данных.
4. Подключите функцию в UI.
5. Проверьте `9-box` и `12-box`, если правило влияет на матрицы.

## Работа С D3 Оргграфом

`d3-org-chart` управляет DOM вне React, поэтому граф изолирован в `components/d3-org-chart-view.tsx`.

Правила сопровождения:

- данные для графа формируйте в `lib/org-chart-d3-data.ts`;
- HTML узлов меняйте в `lib/d3-org-chart-node-html.ts`;
- все динамические строки экранируйте через `escapeOrgChartHtml`;
- не переносите строковые HTML-шаблоны в JSX, пока граф управляется `d3-org-chart`;
- после изменений проверяйте поиск, раскрытие узлов, zoom/fit и drawer деталей.

## Известные Особенности

- Данные моковые и предназначены для локальной демонстрации.
- `web/.next` является сгенерированным артефактом и не должен попадать в git.
- `d3-org-chart` типизирован через локальный `types/d3-org-chart.d.ts`.
- В ESLint могут оставаться warnings по существующим hook dependency и a11y местам; ошибки должны исправляться до merge.
- `skipLibCheck` и `allowJs` включены в `tsconfig.json` как стартовые компромиссы проекта.
